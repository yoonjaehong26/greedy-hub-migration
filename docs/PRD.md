# 그리디 허브 (Greedy Hub) — PRD

> **상태**: v0.5 초안 (티키타카 진행 중) · **작성 목적**: 메인테이너 설득 + 기획 확정
> **한 줄 요약**: 그리디의 *연혁·멤버·스터디·미션·게시판·지원·추억* 을 한곳에서 추적하는 동아리 공식 허브.

## 1. 배경 (그리디는 어떤 동아리이고, 무엇이 불편한가)

### 1.1 그리디 한눈에
- **소속/대상**: 세종대학교. 재학생·휴학생 중 **개발에 관심 있는 누구나**.
- **규모**: 1~4기 누적 약 **50명**. **백엔드·프론트 2트랙** 운영.
- **모집 주기**: 1년 2회(각 방학에 멤버 모집) → 학기 시작과 함께 운영.
- **활동 사이클**: `모집 → OT → 스터디(매주 미션 + 리뷰 티키타카) → 팀 프로젝트(데모데이) → 회고`. 학기 중간중간 친목·학술 행사.
- **기수 병행**: 팀 프로젝트가 **다음 기수 모집과 맞물려** 한 학기를 넘기는 경우가 많아, 짧게나마 **두 기수가 동시에 활성**인 구간이 있다.

**4기 구성 (예시 스냅샷 — 기수마다 인원은 메인테이너·리드가 결정)**

| 트랙 | 메인테이너 | 스터디 리드 | 리뷰어 | 멤버 |
|---|---|---|---|---|
| 공통/운영 | 10 | — | — | — |
| 프론트(FE) | (겸직) | 5 | 6 | 4 |
| 백엔드(BE) | (겸직) | 4 | 9 | 6 |

- **역할은 겸직이 흔하다**: 리드↔리뷰어 병행이 많고, 리드는 대개 메인테이너도 겸한다. → 한 사람이 **여러 역할을 동시에** 가질 수 있어야 한다.

> 역할별 상세 책임·선발 방식은 [4장](#4-사용자--역할), 용어는 [부록 D 용어집](#부록-d-용어집).

### 1.2 문제 정의 (페인포인트)
그리디는 활발히 굴러가지만, **활동의 기록과 소통이 Discord에 휘발되고 한곳에 남지 않는다.** 정보가 디스코드 채널들·트랙별 노션·구글폼·개인 저장소로 흩어져 있고, 기수가 바뀌면 맥락이 사라진다.

| # | 페인포인트 | 지금 | 결과 | 심각도 |
|---|---|---|---|---|
| P1 | **미션·리뷰 트래킹 부재** | 리뷰요청 채널에 PR링크+리뷰어 멘션, 스레드에서 티키타카 | 메인테이너·리드가 진행상황을 **다 추적 못 함**, 완주율 모름 | ★★★ |
| P2 | **스터디 콘텐츠 휘발** | 커리큘럼·미션 요구사항을 디스코드로 공유 | 시간 지나면 묻혀 **재사용·축적 불가** | ★★★ |
| P3 | **리뷰 가이드 전달 비효율** | 리드→리뷰어 가이드라인을 디스코드/구두로 | 리뷰어(이전 기수)가 미션별 리뷰 포인트를 **한눈에 못 봄** | ★★★ |
| P4 | **대외 얼굴·아카이브 없음** | 공식 홈페이지 없음 | 외부에 그리디를 못 알리고, **멤버 취업/구인 때 증빙·포트폴리오로 못 씀** | ★★☆ |
| P5 | **신원·권한 부재** | 누가 몇 기·무슨 역할인지 흩어짐 | 권한별 접근/관리 불가 | ★★☆ |
| P6 | **연혁·추억 휘발** | 추억저장소가 디스코드 채널 | 기수 넘어가면 역사·사진이 소실 | ★★☆ |
| P7 | **지원 관리 수기** | 에타 구글폼 + 수기 서류/면접 | 지원자 상태 추적 분산 | ★☆☆ |

> **핵심 통찰**: 도구를 *추가*하는 게 아니라, 흩어진 것을 **웹 한 층으로 모으고**, 이미 쓰는 도구(GitHub·Discord 봇)는 각자 잘하는 역할만 맡긴다.
>
> **웹 = 기록(진실의 원천) / GitHub = 산출물 / Discord(봇) = 알림·액션**

## 2. 목표 / 비목표

**목표 (이번 범위가 이루려는 것)**
- G1. 미션 출제→제출(GitHub PR)→리뷰→**현황 추적**을 이중작업 없이 한곳에서.
- G2. 연혁·멤버·추억을 **영구 기록**으로 — 기수가 바뀌어도 남는다.
- G3. GitHub 로그인 기반 **신원·기수·권한** 체계.
- G4. 트랙별 스터디 자료·커리큘럼을 검색 가능한 단일 저장소로.
- G5. 지원 접수→심사→멤버 전환을 한 플로우로.
- G6. 지원자·외부에 보여줄 **공개 페이지**(SEO) — 연혁·프로젝트·기술블로그.
- G7. **리뷰 가이드라인을 미션에 축적** — 리뷰어가 한눈에 보고, 다음 기수가 재사용.
- G8. **기술블로그·아카이브 + 멤버 자동 이력서**로 외부 홍보 + 멤버 취업 증빙 + 이전 기수의 지속 기여.

**비목표 (이번엔 안 한다 — 범위 방어)**
- N1. Discord를 대체하지 않는다. (실시간 대화는 계속 Discord)
- N2. 자체 코드 채점/CI 빌드를 처음부터 만들지 않는다. (미션 자동검증은 후속 단계)
- N3. 모바일 네이티브 앱 없음. (반응형 웹으로 충분)
- N4. 결제·회비 정산 시스템 없음. (필요 시 별도 논의)

## 4. 사용자 & 역할

**역할별 실제 책임 (그리디 운영 기준)**

| 역할 | 하는 일 | 선발 방식 |
|---|---|---|
| **메인테이너(Maintainer)** | 동아리 운영 전반, 행사 기획, 대외 연락, 내부 조율·전반 개선. 팀 프로젝트에 **요구사항을 던지고 데모데이** 운영. | 이전 기수 메인테이너가 **컨택**해 다음 기수 구성 |
| **스터디 리드(Lead)** | 트랙 스터디 운영: 매주 미션 + 그 주 마무리에 미션 핵심·리뷰 티키타카 정수를 모아 **대면 스터디** 진행. **커리큘럼 설계·미션 개선**, **리뷰어에게 리뷰 가이드라인 전달**. | 이전 기수 메인테이너 컨택 (대개 메인테이너 겸직) |
| **리뷰어(Reviewer)** | 멤버 미션 제출에 매주 **리뷰·티키타카**. 과거엔 외부/현직자, 현재는 **이전 기수**가 다수 → 미션별 **리뷰 가이드라인이 한눈에 보여야** 한다. | 이전 기수 메인테이너 컨택 (리드와 병행 多) |
| **멤버(Member)** | 스터디를 따라가며 매주 미션 수행·제출, 이후 팀 프로젝트 참여. | 방학에 **에타(에브리타임)에 구글폼** → 1차 서류 → 대면 면접 |

**페르소나**: 지원자/방문자 · 멤버 · 리뷰어 · 리드 · 메인테이너.

**역할은 두 축 + 트랙으로 본다.**

1) **글로벌 역할(서열 고정, 가입 시 부여)** — 권한 베이스라인
```
GUEST  →  MEMBER  →  STAFF
```

2) **기수 역할 (CohortMember — 기수마다, 발자취에 표기)**
```
roles[]: { MEMBER, REVIEWER, LEAD, MAINTAINER }   ← 다중 보유 가능(겸직)
track:   FE | BE | COMMON
```
- 기수 = **입회 기수**(학기당 1, 연 2기, 현재 **4기**). 한 사람이 여러 기수에 다른 역할로 등장(예: 2기 멤버 → 4기 BE 리뷰어+리드).
- **다중 역할**: 한 사람이 같은 기수에서 리드+리뷰어+메인테이너를 동시에 가질 수 있다(겸직이 흔함) → `roles`는 **집합**.
- **트랙**: 멤버·리드·리뷰어·미션·스터디 자료는 FE/BE로 나뉜다. 메인테이너·연혁 등은 COMMON.
- 기수 역할이 권한을 **스코프**: 리뷰어 → 배정 미션에 `mission:review`, 리드 → 트랙 `mission:create`/`study:manage`, 메인테이너 → STAFF급 전반.

세분화된 권한은 **운영진이 UI에서 개별 부여**할 수 있다(아래 RBAC). 이 권한 관리 UI는 **MVP에 포함**.

## 5. 권한 모델 (RBAC)

- **역할(Role)**: 가입 시 부여, 서열 고정 — `GUEST / MEMBER / STAFF`.
- **세부 권한(Permission)**: `mission:create`, `mission:review`, `mission:assign`(리뷰어 배정), `study:manage`, `board:moderate`, `recruit:review`, `gallery:curate`, `project:manage`, `history:edit`, `member:manage`, `admin:roles` …
- **실효 권한 = 글로벌 역할 기본 권한 ∪ 기수 역할이 주는 스코프 권한 ∪ 운영진이 개별 부여한 권한.**
  - 예: 평소 MEMBER지만 4기 리뷰어로 배정되면 그 미션들에 한해 `mission:review` 획득.
  - 예: 한시적으로 출제를 맡기면 `mission:create`만 콕 부여.
- **소유권 규칙**: 역할과 별개로 "본인이 만든 글/제출만 수정" 항상 적용.
- **보안 원칙**: 권한 체크의 진짜 방어선은 **백엔드**. 프론트는 UI 숨김(편의)만 담당.

> 상세 권한 매트릭스는 [부록 B](#부록-b-권한-매트릭스).

## 6. 기능 요구사항

각 기능: *목적 / 페인포인트 / 핵심 기능 / 사용자 스토리 / 권한*.

### 6.1 인증 & 온보딩
- **목적**: GitHub 로그인 한 번으로 신원·기수·권한·깃허브 연동.
- **페인포인트**: P5. 신원·권한 개념 부재, 깃허브-동아리 신원 분리.
- **핵심 기능**
  - GitHub OAuth 로그인. 최초엔 `GUEST`.
  - 프로필에 GitHub username/아바타 자동 연동 (미션 PR 매칭 키).
  - `MEMBER` 승격: 지원 합격 시 자동, 또는 운영진 수동.
- **사용자 스토리**
  - 방문자로서 GitHub로 로그인해 회원 공간에 접근하고 싶다.
  - 운영진으로서 한 멤버의 역할/권한을 바꾸고 싶다.
- **권한**: 로그인 전체 / 역할·권한 변경 `admin:roles`.

### 6.2 미션 (Mission) ⭐ 최우선
- **목적**: 출제→GitHub PR 제출→리뷰어 티키타카→머지(완료)→추적을 **이중작업 없이**.
- **페인포인트**: P1(미션·리뷰 트래킹 부재) + P2(스터디 콘텐츠 휘발) + P3(리뷰 가이드 전달 비효율).
- **현재 흐름(그리디 실제 방식 반영)**:
  1. 멤버가 미션 레포(그리디 또는 **초록스터디 `github.com/cho-log`** — 백엔드는 여기서 진행되기도)를 fork→작업→**PR**.
  2. 디스코드 **리뷰요청 채널**에 PR 링크 + 리뷰어 멘션 → **봇이 매 미션 리뷰어 랜덤 배정**.
  3. 멤버↔리뷰어가 PR/스레드에서 **리뷰 티키타카**(여러 라운드). **정해진 리뷰 기간**, **리뷰어 재량 연장**.
  4. **리뷰 기간이 끝나면 자동 머지** → 종료(=완료). *approve가 곧 통과가 아니라 티키타카가 끝나는 개념.*
  - 트랙(FE/BE)별로 주기·티키타카가 다르며, **시험 몇 주 전·공휴일 주는 미션을 쉰다(SKIPPED)**. 제출이 늦거나 많이 미흡하면 리뷰를 진행하지 않기도 한다.
- **핵심 기능**
  - **미션 정의**: 제목·설명·**요구사항 체크리스트(멤버용)**·트랙·주차/단계 라벨·기본 리뷰 기간·연결 GitHub 레포·상태(`DRAFT/OPEN/CLOSED/SKIPPED`).
  - **⭐ 리뷰 가이드라인(리뷰어용, P3 해결)**: 리드가 미션마다 `reviewGuide`(자유 마크다운 + 선택적 리뷰 체크포인트)를 작성. **리뷰어가 배정 PR 화면에서 항상 같이 봄** → "이 미션 이렇게 리뷰" 한눈에. 미션에 영구 저장되어 **다음 기수 리뷰어가 재사용**(P2 휘발 동시 해결).
  - **제출 매칭**: 멤버는 평소대로 PR을 올리고, 웹은 GitHub username·PR 제목 규칙으로 **그 PR을 자동 매칭/표시**. 보조로 수동 링크 등록 가능.
  - **리뷰어 배정**: 랜덤 배정은 **기존 디스코드 봇 `/리뷰매칭`이 수행**(8.1). 웹은 결과를 **기록**(담당 리뷰어 표시)·필요시 수동 재배정. 명단은 웹의 기수 멤버에서 조회.
  - **리뷰 진행 추적**: 상태(제출됨/리뷰중/완료), **리뷰 마감일(연장 이력)**, 라운드 수, PR 링크. *실제 대화는 GitHub PR, 웹은 상태·기한만 추적.*
  - **리뷰 기간**: **동아리 공통 기본값**(ClubConfig) + 미션별 예외 + 리뷰어 연장.
  - **트래킹 3종**: ① 멤버 "내 미션 현황"(리뷰중/완료/마감임박) ② 운영진 `멤버×미션` 매트릭스 ③ 기수·트랙 완주율 대시보드.
  - **리뷰어 워크스페이스**: 리뷰어가 자신에게 배정된 PR + 각 미션 가이드라인을 한 화면에서.
  - **알림**: 배정·새 리뷰 코멘트·머지(완료)를 Discord 웹훅으로.
  - **(미래) 환류 루프**: 리뷰 티키타카에서 나온 좋은 포인트를 모아 리드의 **대면 스터디 자료**(6.5)로 연결.
- **GitHub 연동 깊이 (단계적)**
  - 1단계(MVP): PR 링크 등록 + 리뷰어 배정·상태·마감일 수동 관리 + Discord 알림. (흩어진 흐름을 "기록되는 버전"으로 이전)
  - 2단계: GitHub API로 PR 상태·리뷰 코멘트 수·머지 여부를 자동 동기화.
  - 3단계: 리뷰 기간 종료 시 **자동 머지 트리거**, 요구사항 일부 자동검증.
- **사용자 스토리**
  - 멤버로서 내가 리뷰 중인/마감 임박한 미션을 한눈에 보고 싶다.
  - 리뷰어로서 내게 배정된 PR과 리뷰 마감일을 모아 보고, 필요시 기간을 연장하고 싶다.
  - 운영진으로서 기수 완주율과 누가 어느 단계를 빠뜨렸는지 매트릭스로 보고 싶다.
- **권한**: 출제·요구사항 `mission:create` / 리뷰어 배정 `mission:assign` / 리뷰·기간연장 `mission:review`(배정된 리뷰어) / 제출·조회 `MEMBER`.
- **상세 상태머신**: [부록 C](#부록-c-미션-상태머신).

### 6.3 연혁 (History / Timeline)
- **목적**: 그리디가 걸어온 길 — 공개 페이지로 보여줄 얼굴.
- **페인포인트**: P6. 연혁·추억 휘발.
- **핵심 기능**: 연도/기수별 타임라인, 이벤트 카드(제목·날짜·사진·설명), 공개 열람.
- **사용자 스토리**: 방문자로서 그리디의 역사를, 운영진으로서 새 이벤트를 추가하고 싶다.
- **권한**: 열람 전체 / 작성·수정 `history:edit`.

### 6.4 멤버 (Members / 발자취 = 자동 이력서) ⭐
- **목적**: 멤버 프로필을 **자동으로 쌓이는 이력서/포트폴리오**로. 별도 설명 없이 `/members/[username]` 하나로 "이 사람이 그리디에서 뭘 했는지"가 드러나, 취업·구인 증빙(P4·G8)으로 바로 쓰인다.
- **페인포인트**: P4(취업 증빙)·P5(신원)·P6(휘발). 활동이 디스코드·PR에 흩어져 **개인 이력으로 안 남음**.
- **핵심 기능**
  - 공개 프로필: 이름·깃허브·입회기수·한줄소개·링크.
  - **활동 이력(자동 누적)** — 미션 하나 끝날 때마다 자동 반영, **기수가 넘어가도 영구 보존**:
    - 완료 미션 목록(미션명·트랙·기수·PR 링크·리뷰 완료 여부),
    - **기수별 역할 발자취**(예: "2기 멤버 → 4기 BE 리뷰어+리드"),
    - 작성한 기술블로그(6.6), 참여한 팀 프로젝트(6.9).
  - **이력서 뷰**: 공개 URL로 공유 가능. (선택) 본인이 노출/하이라이트 항목 선택.
  - 기수·트랙별 멤버 디렉토리.
- **데이터**: 별도 큰 테이블 없이 `Submission(MERGED)`·`CohortMember.roles`·`Post(BLOG)`·`Project`를 **user 기준 집계 뷰**로 구성. (선택) 핀 항목 `ProfilePin`.
- **초기 백필**: [10.1 데이터 이행](#101-데이터-이행-백필) 참고 — 홈페이지 이전 이력은 깃허브 PR에서 반자동 임포트 후 자동 누적으로 전환.
- **사용자 스토리**: 멤버로서 내 활동이 **자동으로 이력서에 쌓이길** 원한다 / 본인 프로필 편집 / 운영진의 기수 지정·멤버 관리.
- **권한**: 공개 프로필 열람 전체 / 본인 수정 / 관리 `member:manage`.

### 6.5 스터디 (Study)
- **목적**: 트랙별(FE/BE) 커리큘럼 + 자료 단일 저장소. (현재 트랙별 노션에 흩어진 것을 모음/링크)
- **페인포인트**: P2. 스터디 콘텐츠 휘발·분산.
- **핵심 기능**: **트랙별** 주차/단계 커리큘럼, 자료(링크·파일·**노션 externalUrl**) 업로드, 검색·태그. 대면 스터디에서 다룬 핵심·리뷰 정수 정리.
- **사용자 스토리**: 멤버의 자료 열람 / 리드의 커리큘럼·자료 편집.
- **권한**: 멤버 열람+업로드 / 커리큘럼 편집 `study:manage`(리드).

### 6.6 기술블로그 & 게시판 (Blog & Board)
- **목적**: 공지/자유 글 + **기술블로그**(멤버·이전 기수가 쓰는 기술 글) 아카이브.
- **페인포인트**: P4(대외 아카이브 없음) + 공지가 Discord에 묻힘. 스터디 끝난 **이전 기수가 기여할 통로**가 없음.
- **핵심 기능**
  - 카테고리: **기술블로그** / 공지 / 자유.
  - 글·댓글 CRUD(마크다운·코드블록), 태그·검색, 공지 고정.
  - **기술블로그는 공개(SEO)** → 외부에 그리디 활동을 보여주고, 멤버 **취업/포트폴리오 증빙**으로 활용. 작성자 프로필·기수 발자취와 연결.
  - 이전 기수(졸업 멤버)도 작성 가능 → 지속 기여.
- **사용자 스토리**: 멤버·이전 기수의 기술 글 작성 / 운영진의 공지·모더레이션 / 방문자의 공개 블로그 열람.
- **권한**: 공개 글 열람 전체 / 작성·댓글 `MEMBER`(이전 기수 포함) / 모더레이션 `board:moderate` / 본인 글만 수정.

### 6.7 지원 (Recruit)
- **목적**: 비회원 지원→**단계 심사(서류→면접→합격)**→멤버 전환을 한 플로우로.
- **페인포인트**: P7. 에타 구글폼+수기, 상태 추적 분산.
- **핵심 기능**
  - **웹 자체 지원 폼**(모집기간 노출) → 응답이 곧장 웹 `Application`으로 저장. (구글폼 의존 탈피. 기존 봇의 구글폼 감시는 설문 등 다른 용도로 잔존 가능.)
  - 본인 지원 현황 조회.
  - 운영진 **심사 보드**: 단계 칸반(서류 → 면접 → 합격/불합격), 단계 이동·코멘트.
  - 합격 시 `MEMBER` 승격 + 해당 기수 `CohortMember` 생성 연동.
- **단계 상태**: `SUBMITTED → DOC_PASS → INTERVIEW → ACCEPTED | REJECTED`.
- **사용자 스토리**: 방문자의 지원 / 운영진의 단계별 심사·합격 처리.
- **권한**: 제출 `GUEST` / 본인 조회 / 심사 `recruit:review`.

### 6.8 추억 저장소 (Gallery)
- **목적**: 사진·영상·회고 아카이브.
- **페인포인트**: P6. 추억저장소가 디스코드 채널 → 기수 넘어가면 소실.
- **핵심 기능**: 앨범(이벤트·기수별), 미디어 업로드, 회고 글, 공개/회원전용 구분.
- **사용자 스토리**: 멤버의 업로드 / 운영진의 큐레이션.
- **권한**: 공개 갤러리 열람 전체 / 업로드 `MEMBER` / 큐레이션·삭제 `gallery:curate`.

### 6.9 프로젝트 / 데모데이 (Projects) — *MVP는 아카이브 수준*
- **목적**: 스터디 이후 진행되는 **기수별 팀 프로젝트**를 보여주는 아카이브. (대외 홍보·포트폴리오 가치)
- **배경(그리디 운영)**: 스터디가 끝나면 팀을 꾸려 프로젝트 진행. 메인테이너가 프로젝트에 **요구사항을 던지고 `데모데이`에서 발표**. 기존 깃허브에 기수별 팀 프로젝트 저장소가 있다.
- **MVP 범위(아카이브)**: 기수·트랙별 **프로젝트 카드**(팀명·멤버·한줄 소개·GitHub/배포 링크·썸네일). 공개 열람.
- **추후 확장(메모, 이번엔 미구현)**:
  - **데모데이** 일정·회차 관리, **데모데이별 요구사항**(메인테이너가 팀에 던지는 요구사항)을 미션처럼 등록·추적.
  - 발표 자료·피드백 기록, 데모데이 결과 아카이브.
- **권한**: 공개 열람 / 프로젝트 등록·수정 `project:manage`(메인테이너) 또는 본인 팀.

## 8. 시스템 아키텍처

- **프론트엔드**: Next.js (App Router) — 공개 페이지 SSR/SEO, 회원/운영진은 인증 기반 라우팅. TypeScript + Tailwind.
- **백엔드**: 운영진이 별도 언어로 구현(Java/Python/Go 등).
- **계약(Contract)**: **REST + OpenAPI 단일 스펙(`openapi.yaml`)**.
  - 백엔드가 스펙을 단일 소스로 관리 → 프론트는 `openapi-typescript`로 **타입·클라이언트 자동생성**.
  - 프론트는 스펙 기반 **목 서버**로 백엔드 없이 선행 → 병렬 개발.
- **인증**: GitHub OAuth (백엔드가 토큰 교환·세션 발급).
- **연동**: GitHub API(**`greedy-team` + `cho-log` 두 오거 모두**의 미션 PR/커밋 조회, 단계적) · **기존 `greedy-discord-bot`(아래 8.1)** · Discord 웹훅(알림).
- **스토리지**: 미디어(추억/자료)는 오브젝트 스토리지(S3 호환) + DB엔 메타데이터.
- **배포**: 프론트 Vercel, 백엔드 운영진 택. CI에서 타입체크·빌드·OpenAPI 검증.

```
[Next.js]  ⇄  openapi.yaml (단일 계약)  ⇄  [백엔드(別언어)]
    │              │                            │
    │        타입 자동생성        GitHub API / greedy-discord-bot / Discord Webhook
    └─ 목 서버로 선행개발                        └─ DB · Object Storage
```

### 8.1 기존 디스코드 봇과의 통합 경계 (중요)
그리디는 이미 `greedy-team/greedy-discord-bot`(Java/JDA)을 운영 중. **재구현하지 않고 통합**한다.

봇이 이미 하는 것: **리뷰어 랜덤 매칭**(`/리뷰매칭`, `MatchingService`+`RandomShuffleStrategy`), **구글폼 응답 감시**(`GoogleFormWatch`), 예약 메시지, 정산, 운세, 스터디룸, 디스코드 역할 관리.

**현재 한계(=웹이 메울 부분)**
- 매칭 결과가 `toDiscordAnnouncement()` 문자열로 **디스코드 공지만** 되고 **영구 기록이 없다** → 완주율·발자취 추적 불가(P1의 근원).
- 리뷰이/리뷰어 명단이 봇 코드에 **하드코딩** → 매 라운드 손수정.

**통합 원칙: 봇=액션, 웹=기록.**
| 흐름 | 주체 | 내용 |
|---|---|---|
| 명단 제공 | 웹 → 봇 | 봇이 매칭 시 리뷰이/리뷰어 풀을 **웹 API에서 조회**(기수 멤버·`roleInCohort=REVIEWER`). 하드코딩 제거. |
| 매칭 기록 | 봇 → 웹 | `/리뷰매칭` 확정 시 결과를 **웹 API로 POST** → `ReviewAssignment`로 영구 저장. |
| 지원 수집 | 봇 → 웹 | 구글폼 응답 감지 시 **웹 API로 전달** → `Application` 생성(또는 웹 자체 폼으로 대체). |
| 알림 | 웹 → 봇/웹훅 | 제출·리뷰·머지 이벤트를 디스코드로 통지. |

> 효과: 봇의 검증된 기능은 그대로 두고, **휘발되던 결과를 웹이 역사로 남긴다.** 하드코딩 유지보수도 소멸.

## 9. MVP 범위 & 로드맵

원칙: **한 번에 다 만들지 않는다.** 가치가 가장 큰 미션 추적부터.

- **MVP (1차) — "로그인하면 미션·리뷰가 추적되고, 멤버 이력서가 자동으로 쌓인다"**
  - 인증(GitHub) + 역할(글로벌 3단계 + 기수 역할/트랙/다중역할) + **권한 세분화 관리 UI** + 미션(정의·**리뷰 가이드라인**·PR 링크 등록·리뷰어 배정 기록·상태/마감일·현황 매트릭스) + **리뷰어 워크스페이스** + **멤버 자동 이력서**(+초기 백필) + 연혁 + 공개 홈.
- **2차 — 스터디 & 기술블로그**
  - 트랙별 스터디 자료/커리큘럼(노션 링크 포함), **기술블로그**(공개·SEO·이전 기수 기여).
  - 미션 GitHub 연동 2단계(PR 상태·리뷰 코멘트 자동 동기화, `greedy-team`+`cho-log`).
- **3차 — 지원 & 프로젝트 아카이브**
  - 웹 자체 지원 폼·심사 보드·멤버 전환. **프로젝트 아카이브**(기수·트랙별 카드).
- **4차 — 추억 저장소 & 자동화 & 데모데이**
  - 미디어 업로드·앨범, 리뷰 기간 종료 시 자동 머지 트리거, 요구사항 자동검증(3단계), **데모데이·데모데이별 요구사항**.

## 10. 비기능 요구사항
- 반응형 + **다크모드**, 웹 접근성 기본 준수.
- 권한 서버 강제(프론트는 편의 숨김만).
- OpenAPI 단일 계약 → 타입 자동생성·목 서버 병렬개발.
- 성능: 공개 페이지 SSR/캐싱으로 빠른 첫 화면.
- 관측: 에러 로깅·기본 분석.

### 10.1 데이터 이행 (백필)
홈페이지 이전의 활동도 멤버 이력서(6.4)에 남아야 한다. **출시 시점 1회 백필 → 이후 자동 누적**.
- **반자동 임포트**: 과거 제출은 이미 `greedy-team`·`cho-log` 레포의 **PR**로 존재(제목 규칙 `[단계] 이름 미션 제출`). PR 히스토리를 긁어 **author 깃허브 username으로 멤버 매칭** → 미션·`Submission(MERGED)` 레코드 생성.
- **수동 보정**: 규칙을 벗어난 과거 PR·기수/트랙 매핑·외부 진행분은 운영진이 보정.
- **전환**: 백필 완료 후 신규 제출은 자동 기록(설명 불필요).
- 범위: 최소 멤버↔미션 매핑(누가 무엇을 완료)부터. 리뷰 상세까지는 선택.

## 11. 디자인 방향 (미정 — 동아리와 합의)
- 동아리 **공식 브랜딩**은 기존 개인 서비스 톤을 참고하지 않고, 그리디 정체성으로 별도 합의 후 확정.
- 본 PRD는 기능·구조에 집중하고, 비주얼/브랜딩은 별도 디자인 트랙에서 진행.

## 12. 리스크 & 가정
- **R1. FE/BE 사람이 다름** → 계약(OpenAPI) 미합의 시 재작업 위험. *완화*: 코드 전에 스펙·목 서버부터.
- **R2. 미션 PR 자동매칭 정확도** → username·제목 규칙 의존. *완화*: 1단계는 수동 등록 허용, 규칙 문서화.
- **R3. 운영 인력 지속성** → 유지보수 부담. *완화*: 관리 화면 단순화, MVP 범위 엄수.
- **R4. 개인정보(지원서)** → 수집·보관 동의 필요.
- **R5. 봇↔웹 계약** → 봇 명단을 웹에서 조회/결과 POST하도록 봇 수정 필요(봇 운영자와 합의). *완화*: 봇 변경 없이 "수동 기록"으로도 동작하게 설계, 통합은 단계적.
- **R6. 백필 정확도**(10.1) → 과거 PR 제목 규칙 이탈·username 변경으로 매칭 누락. *완화*: 반자동 임포트 후 운영진 수동 보정, 멤버 본인 확인.
- **가정**: 회원 대부분 GitHub 계정 보유, Discord를 일상적으로 사용, 봇 운영자와 협업 가능.

## 부록 C. 미션 상태머신
```
[미션]   DRAFT ── 공개 ──> OPEN ── 마감 ──> CLOSED

[제출]   (없음)
           │ 멤버가 PR 제출
           ▼
        SUBMITTED ── 리뷰어 랜덤 배정 ──> IN_REVIEW
                                          │  멤버↔리뷰어 티키타카 (N라운드)
                                          │  reviewDueAt = 기본기간, 리뷰어가 연장 가능(extendedCount++)
                                          ▼  리뷰 기간 종료 → 자동 머지
                                        MERGED (완료)
```
- *approve는 종료 신호가 아니라 대화의 일부.* 완료 판정 = **리뷰 기간 종료에 따른 머지**.
- 알림 트리거: SUBMITTED → 배정된 리뷰어 / 새 리뷰 코멘트 → 상대방 / MERGED → 멤버.

## 부록 D. 용어집 (Glossary)
- **트랙(Track)**: 그리디의 두 갈래 — **FE(프론트)** / **BE(백엔드)**. 멤버·리드·리뷰어·미션·스터디 자료가 트랙별로 나뉜다.
- **스터디**: 매주 미션을 수행하고, 그 주 마무리에 미션 핵심·리뷰 티키타카 정수를 모아 **대면으로 모여** 진행하는 트랙별 학습 활동.
- **미션**: 매주(시험·공휴일 주는 쉼) 주어지는 과제. 보통 4~5일 구현 + 3~4일 리뷰 티키타카(다음 미션과 겹치기도 함).
- **리뷰 티키타카**: 멤버 PR에 대해 리뷰어가 코멘트하고 멤버가 반영·재질문하며 주고받는 리뷰 과정. approve가 아니라 **티키타카 종료**가 마무리.
- **리뷰 가이드라인**: 리드가 미션마다 작성해 리뷰어에게 주는 "이 미션은 이런 포인트로 리뷰하면 좋다" 지침.
- **데모데이**: 메인테이너가 팀 프로젝트에 요구사항을 던지고, 팀이 구현해 발표하는 정기 행사.
- **초록스터디(cho-log)**: 그리디가 예하로 참여하는 외부 스터디. 백엔드 미션·리뷰가 여기 디스코드/깃허브(`github.com/cho-log`)에서 이뤄지기도 한다.
- **에타(에브리타임)**: 멤버 모집 구글폼을 올리는 대학 커뮤니티.
- **기수(Cohort)**: 입회한 모집 회차(예: 4기). 학기당 1기수, 연 2기수. 프로젝트가 다음 기수와 맞물려 **잠시 2기수 동시 활성**.
